Overview

Your setup is intentionally sparse. That is a strength. You are adding capability through a small number of sharp extensions, not by turning every knob in settings.json.

What is already good

Minimal defaults, sensible model choice, hidden thinking block, and a small MCP surface instead of a bloated tool list.

Where the real issues are

Package reproducibility, runtime-state churn in tracked settings, and the operational weight of the browser tools extension.

What I would not change casually

The overall philosophy. This config is opinionated in a good way: focused, low-noise, and easy to reason about.

Findings

Ranked by practical impact. None of these are emergencies. The first two are the highest-leverage cleanup items.

Medium

Pin the MCP adapter source instead of tracking GitHub main implicitly

In pi/.pi/agent/settings.json you load the adapter via a raw GitHub URL. That works, but it is less reproducible than a pinned npm package or a git ref.

Current source: https://github.com/nicobailon/pi-mcp-adapter
Installed package version: 2.4.0
  • The adapter's own README recommends pi install npm:pi-mcp-adapter.
  • Using a tag or explicit npm version makes machine rebuilds and future audits much clearer.
  • If you like following head intentionally, keep it — but that should be a conscious choice, not an accident.
Medium

lastChangelogVersion looks like machine state, not durable config

This value is present inside your tracked settings.json. That means normal pi updates can create dotfiles churn for something that is really just “what version has already shown me the changelog?”

  • It is not harmful at runtime.
  • It is noisy in a dotfiles repo because it will change over time and may differ across machines.
  • If you keep syncing the full file, expect recurring diffs from this key.
Strength

The core configuration is refreshingly minimal

Your top-level settings define only what clearly matters: provider, model, thinking level, padding, theme, package source, and whether to hide thinking output. That is a better maintenance posture than an over-tuned settings file full of speculative options.

  • defaultThinkingLevel: "high" + hideThinkingBlock: true is a coherent pair.
  • editorPaddingX: 1 is a tasteful tiny UX adjustment rather than meaningless theme churn.
  • The setup is legible at a glance.
Low

The browser tools extension is powerful, but heavy for a dotfiles repo

The local browser-tools package follows pi’s documented “package with dependencies” extension pattern correctly — but the dependency footprint is large.

  • pi/.pi/agent/extensions/browser-tools is about 198 MB.
  • Its vendored node_modules contains 421 directories.
  • That is defensible for a self-contained workstation setup, but not especially elegant for dotfiles.
Low

One small code-quality mismatch exists inside browser-tools

The recording schema says the format defaults to mp4 only if ffmpeg is available, otherwise webm. The implementation currently hardcodes mp4 and only fails later if ffmpeg is missing.

  • This is minor, but it is the clearest correctness nit I found.
  • If you keep this extension long-term, I would fix the behavior or the description so they agree.
Strength

Your prompt-template → skill pattern is good

prompts/evaluation-package.md is a thin, useful shortcut that delegates the real behavior to a proper skill. That separation is clean: command ergonomics in the prompt, durable workflow guidance in the skill.

Optional

A few additions could improve ergonomics, but only if they match your workflow

I do not think you are missing anything essential. These are “nice if you want them,” not “you forgot this.”

  • enableInstallTelemetry: false if you want an explicit privacy stance.
  • enabledModels if you regularly cycle among a small curated set of models.
  • settings.autoAuth: true in mcp.json if Supabase OAuth friction annoys you.
  • Selective directTools for a tiny set of favorite Xcode/Supabase tools if repeated search overhead bothers you.
Context

I would keep defaultThinkingLevel: "high" unless speed/cost is a pain point

This is the one setting that many people would reflexively “optimize” down to medium. I would only do that if you personally feel the latency or usage cost. Otherwise, your current choice is coherent with a quality-first coding workflow.

Risks, caveats, and unresolved items

Places where I am deliberately cautious instead of over-prescriptive.

I did not audit credentials

auth.json, environment variables, and any secret management setup were out of scope.

I did not runtime-test every custom tool path

This was a configuration/code audit, not a full behavioral test pass of ask/webfetch/browser automation.

Some “improvements” are preference-sensitive

Thinking level, direct MCP tool exposure, and telemetry stance depend on how you personally use pi.

Recommended next actions

If you want the highest value from the least effort, I would do these in roughly this order.

Priority 1
Pin the MCP adapter package source

Switch from a floating GitHub URL to either npm:pi-mcp-adapter@2.4.0 or a git URL with an explicit ref. This gives you cleaner rebuilds and more predictable updates.

Priority 2
Decide how you want to handle lastChangelogVersion churn

If tracked settings diffs annoy you, treat this key as machine state and keep it out of your durable config workflow as much as possible.

Priority 3
Clean up browser-tools packaging if repo weight bothers you

Options: keep only the source + package.json and install after stow, move it into a standalone pi package, or keep the current self-contained setup if convenience matters more than repo neatness.

Optional
Add small workflow niceties only if you will use them

Candidates: enableInstallTelemetry: false, curated enabledModels, mcp.settings.autoAuth, or selective directTools for your most-used MCP actions.

Optional
Fix the browser recording default inconsistency

Either probe for ffmpeg and fall back to webm, or change the schema text so it matches the actual behavior.

Bottom line: I would describe this as a good, tasteful pi setup with a couple of infrastructure-level cleanups to make it easier to maintain over time. I do not think you are missing major features.
