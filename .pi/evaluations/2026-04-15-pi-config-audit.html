Strengths

What is already working well

1
You are using Pi's native layout instead of fighting it.
Extensions live in ~/.pi/agent/extensions/, prompts in ~/.pi/agent/prompts/, MCP config in ~/.pi/agent/mcp.json. That keeps settings.json small and means reload / discovery work the way Pi expects.
2
Your custom tools solve real gaps.
ask.ts adds a polished multi-question UI, and webfetch.ts gives Pi a proper fetch primitive with text / html / raw modes. These are pragmatic additions, not novelty baggage.
3
Your MCP setup is restrained.
Only two servers, no giant direct-tool explosion, and no redundant server zoo. That matches the adapter's main benefit: small context footprint, lazy discovery, and less startup clutter.
4
The evaluation-package prompt wrapper is a nice quality-of-life touch.
It turns a skill you use often into a lightweight reusable prompt entrypoint without duplicating the actual skill instructions.
Bottom line: I would preserve the overall shape. This is already a minimal-but-capable config, not a cleanup disaster.
Recommendations

Highest-value changes, ranked

High value

Pin the MCP adapter

Your package source is an unpinned GitHub URL. Pi docs explicitly note that versioned npm specs and git refs are pinned and skipped by pi update. For dotfiles, reproducibility matters.

High value

Turn on MCP autoAuth

Your Supabase server uses OAuth. The adapter can automatically launch auth on connect / tool call and retry once. That is a clean friction-reducer with very little downside.

Medium value

Promote a few direct tools

Because you already have cached metadata for both servers, you can expose only the MCP tools you reach for constantly and keep the rest behind the proxy.

Medium / optional

Revisit global high thinking

If you intentionally want maximum effort by default, keep it. If not, this is the likeliest source of unnecessary latency / cost. A global default of medium plus manual bumps is usually a saner default.

Hygiene

Stop treating lastChangelogVersion as config

That looks like app-managed state, not a preference. It will create dotfile churn whenever Pi updates. I would not consider it part of the canonical config you care about reviewing.

MCP tuning

Keep the proxy model, but expose the tools you actually use

Why this is worth doing

The adapter README says each direct tool costs roughly 150-300 tokens in the system prompt, so this should be selective, not global.

  • Supabase cache already knows about 29 tools
  • Xcode cache already knows about 20 tools
  • You can keep the single proxy tool for breadth
  • And only elevate the tiny handful you use constantly

Reasonable first candidates

Supabase
  • execute_sql
  • apply_migration
  • generate_typescript_types
  • get_logs
Xcode
  • BuildProject
  • GetBuildLog
  • XcodeRead
  • XcodeWrite
  • XcodeGrep
I would not remove pi-mcp-adapter. Pi's own README still says there is no built-in MCP and recommends either CLI tools or an extension. In your setup, the adapter is doing real work.
Settings tuning

Two real tweaks, one taste call, one low-stakes convenience feature

01Reproducibility

Pin the adapter version. This is the clearest hard recommendation.

02Friction

Add autoAuth for Supabase. Tiny config, noticeable convenience.

03Cost / speed

Consider defaultThinkingLevel: "medium" if high is more habit than intent.

04Workflow

Add enabledModels if you frequently switch between a fast model and a deep-reasoning model.

Settings I would leave alone

  • editorPaddingX: 1 — harmless and tidy
  • theme: "dark" — default, fine
  • hideThinkingBlock: true — coherent if you dislike transcript noise

Feature you may want later

If you wind up maintaining more local Pi resources, packaging your own dotfiles Pi resources as a local package could be nice. Right now it would be overkill. Your current structure is simpler.
